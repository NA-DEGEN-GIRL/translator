# Meta Seamless Local Implementation Notes

## 결론

`Meta Seamless` 계열은 로컬에서 구동 가능하다.
특히 현재 PC가 `RTX 5090` 급이고 `WSL2` 사용 가능하다면 하드웨어 자체는 충분히 유리하다.

다만 이 경로는 `OpenAI Realtime API`보다 구현 난이도가 확실히 높다.

이유는 간단하다.

- OpenAI Realtime은 모델 운영을 OpenAI가 대신한다.
- Meta Seamless는 모델 설치, 추론 서버, 스트리밍 처리, 오디오 파이프라인을 직접 만들어야 한다.

즉, 병목은 GPU보다 "시스템 통합"에 가깝다.

## Meta Seamless가 의미하는 것

실제로 검토 대상은 보통 아래 계열이다.

- `SeamlessM4T`
  - speech-to-speech
  - speech-to-text
  - text-to-speech
  - text-to-text
- `SeamlessStreaming`
  - 스트리밍 번역 지향
- `Seamless`
  - expressive + streaming 통합 계열

실시간 통역 관점에서는 `SeamlessStreaming` 또는 최신 통합 `Seamless` 쪽을 먼저 봐야 한다.

## 하드웨어 판단

현재 전제:

- GPU: `RTX 5090`
- OS: Windows + `WSL2`

이 조건이면 로컬 추론 하드웨어는 충분히 강하다.

현실적인 해석:

- "모델이 돌아가냐?" -> 거의 그렇다
- "PoC 수준 실시간 통역이 가능하냐?" -> 가능성 높다
- "바로 제품 수준으로 되냐?" -> 별개 문제

주의점:

- 진짜 병목은 GPU보다 `WSL 오디오 입출력`
- 설치보다 더 어려운 부분은 `실시간 스트리밍 파이프라인`

## 로컬 환경 전제

Meta Seamless 저장소 기준으로 필요한 전제는 대략 아래다.

- `Linux x86-64` 권장
- 또는 `Apple Silicon Mac`
- Python 가상환경
- `ffmpeg`
- `libsndfile`
- `fairseq2` 설치 가능 환경

WSL2는 사실상 Linux로 취급할 수 있으므로 방향은 맞다.

다만 실제 안정성은 아래 조합이 좌우한다.

- Windows NVIDIA 드라이버
- WSL GPU 연동
- PyTorch CUDA 버전
- `fairseq2` 호환 버전
- 오디오 캡처/재생 방식

## 구현 난이도 평가

### 총평

구현 난이도는 `높음`.

비교:

- `OpenAI Realtime API`: 중
- `Meta Seamless local`: 높음

### 왜 어려운가

#### 1. 모델 설치와 런타임 정합성

단순한 pip 설치로 끝나지 않을 수 있다.

- PyTorch CUDA 버전 정합성
- `fairseq2` 설치
- 모델 가중치 다운로드
- 추론 스크립트 검증

즉, "설치 완료"와 "실시간 서비스 가능" 사이에 거리가 있다.

#### 2. 스트리밍 처리

오디오 파일 번역은 비교적 쉽다.
실시간 통역은 전혀 다른 문제다.

필요한 것:

- 마이크 입력 chunking
- chunk별 추론
- 발화 종료 감지
- 번역 음성 출력 타이밍 제어
- 누적 지연 최소화

여기서부터 일반 데모와 제품형 구현이 갈린다.

#### 3. 오디오 파이프라인

현재 앱은 브라우저 기반이다.
즉 로컬 모델을 붙이려면 아래 경로를 만들어야 한다.

`브라우저 마이크 -> 서버 -> 모델 추론 -> 브라우저 스피커`

이때 결정해야 할 것:

- 오디오를 websocket으로 보낼지
- chunk 업로드 방식으로 보낼지
- 음성 결과를 wav/mp3/blob 중 무엇으로 보낼지
- partial transcript를 보여줄지

#### 4. 실시간 UX 유지

지금 앱은 양쪽 화면이 나뉜 2인 대면 UX다.

Meta Seamless를 붙이더라도 아래 문제는 그대로 남는다.

- 한 기기에서 양쪽 음성을 어떻게 받을지
- 누가 말했는지 어떻게 구분할지
- 원문/번역문 버블을 어떻게 동기화할지
- 음성 피드백 루프를 어떻게 막을지

즉 모델을 바꿔도 UX 설계 문제는 자동으로 해결되지 않는다.

#### 5. 유지보수

OpenAI API는 모델 운영을 외부가 담당한다.
Meta 로컬 추론은 네가 운영자가 된다.

필요한 것:

- 모델 업데이트 판단
- 메모리/속도 모니터링
- 장애 복구
- 로깅
- 서비스 프로세스 관리

## 현재 프로젝트에 붙이는 구조

가장 현실적인 구조는 아래다.

### 권장 구조

1. `WSL2` 안에서 Python 추론 서버 실행
2. Windows 브라우저에서 현재 웹앱 사용
3. 브라우저가 마이크 오디오를 서버로 전송
4. 서버가 Meta Seamless 추론 수행
5. 서버가 번역 텍스트/음성을 브라우저로 반환
6. 브라우저가 자막과 음성을 재생

이 구조를 권장하는 이유:

- WSL 내부에서 마이크/스피커를 직접 다루는 것보다 단순함
- 현재 웹앱 구조를 완전히 버리지 않아도 됨
- 디버깅이 더 쉬움

### 비권장 구조

- WSL 내부에서 직접 마이크 입력과 스피커 재생을 모두 처리

이 방식은 오디오 장치 연동 때문에 더 귀찮아질 가능성이 높다.

## 구현 방식 선택지

### 선택지 A: 비실시간 PoC

가장 쉬운 시작점.

- 사용자가 짧은 음성 파일 업로드
- 서버가 Meta Seamless로 번역
- 결과 텍스트와 음성 반환

장점:

- 설치 검증에 좋음
- 모델 자체 성능 확인 가능

단점:

- 실시간 통역 UX 검증 불가

### 선택지 B: 준실시간 청크 처리

- 브라우저에서 오디오를 짧게 잘라 전송
- 서버가 chunk 단위로 처리
- 최종 결과를 순차 출력

장점:

- 현재 앱 구조를 덜 깨고 붙일 수 있음
- 실시간에 가까운 체감 확인 가능

단점:

- chunk 경계에서 어색함
- 지연 튜닝 필요

### 선택지 C: 진짜 스트리밍

- websocket 기반 오디오 스트리밍
- 서버가 스트리밍 입력 처리
- partial transcript / partial translation / audio output 순차 전송

장점:

- 가장 통역기답다

단점:

- 구현 난이도 가장 높음
- 디버깅 비용 큼

## 단계별 현실적 접근

### Phase 1. 설치 검증

- 저장소 clone
- 가상환경 구성
- `ffmpeg`, `libsndfile` 설치
- GPU 인식 확인
- 샘플 음성 파일로 inference 성공 확인

목표:

- "이 환경에서 모델이 정상 추론되는가" 확인

### Phase 2. 서버화

- FastAPI 또는 별도 Python 서버 작성
- 파일 업로드 기반 API 구현
- 텍스트/음성 반환 형식 정의

목표:

- "웹앱에서 호출 가능한 서비스 형태" 만들기

### Phase 3. 준실시간

- 브라우저 오디오 chunk 업로드
- chunk 단위 응답 처리
- 자막 UI 연결

목표:

- "실시간에 가까운 UX" 검증

### Phase 4. 진짜 스트리밍

- websocket 오디오 스트리밍
- partial 결과 처리
- 끼어들기, 턴 전환, 세션 상태 관리

목표:

- "실시간 통역기 수준"으로 고도화

## 리스크

### 1. 설치 리스크

가장 먼저 겪을 수 있는 문제.

- `fairseq2` 호환성
- CUDA/PyTorch 충돌
- WSL 환경 차이

### 2. 지연시간

모델이 돌아간다고 해서 실시간 통역이 되지는 않는다.
특히 speech-to-speech는 텍스트 모델보다 무겁다.

### 3. 오디오 품질

브라우저 마이크 품질, 샘플레이트, chunk 크기에 따라 성능 체감이 크게 달라질 수 있다.

### 4. 2인 대면 UX

현재 앱은 상하 분할 화면 구조다.
실제 한 기기에서 양쪽 화자를 동시에 자연스럽게 처리하는 것은 모델보다 UX와 오디오 설계 문제다.

### 5. 라이선스

Meta Seamless 계열은 상업 사용 전 반드시 라이선스를 별도 검토해야 한다.
기술적으로 가능하다고 바로 서비스에 써도 되는 것은 아니다.

## 현재 앱 기준 판단

### 장점

- 강한 GPU가 있다
- WSL 사용 가능하다
- 이미 웹앱 구조가 있어서 추론 서버만 연결해도 PoC 가능하다

### 단점

- 현재 앱은 OpenAI API 기반 구조에 맞춰져 있다
- 실시간 스트리밍용 서버/클라이언트 프로토콜이 없다
- 음성 입력 상태 관리가 아직 단순하다

즉, Meta Seamless를 붙이는 건 가능하지만 "모델만 갈아끼우는 작업"은 아니다.

## 추천 판단

### Meta Seamless를 할 만한 경우

- 로컬 모델 실험이 중요하다
- 벤더 종속을 줄이고 싶다
- GPU 자원이 충분하다
- 설치/운영 복잡도를 감수할 수 있다

### OpenAI Realtime이 더 나은 경우

- 빠르게 제품 체감을 만들고 싶다
- 구현 시간을 줄이고 싶다
- 모델 운영 부담을 피하고 싶다

## 최종 요약

현재 장비라면 `Meta Seamless local`은 충분히 시도할 가치가 있다.
하지만 난이도는 명확히 높다.

실제 일의 대부분은 다음에 있다.

- 모델 설치
- 추론 서버화
- 브라우저 오디오 스트리밍
- 자막/음성 동기화
- 실시간 상태 관리

즉, 이 경로는 "모델 선택" 문제가 아니라 "실시간 음성 시스템 구현" 문제다.
